# 特殊設計：題目溯源與成績雙向同步機制

## 解決的核心問題

在考試系統中，管理者經常需要從題庫中**抽題組卷**——例如從「工業安全」的各章節中挑選題目，組成一份「模擬考第一回」。

傳統做法下，學員考完模擬考後會遇到一個致命問題：

> **「我這題錯了，但它原本屬於哪個章節？我哪裡不會？」**

學員無法從模擬考的成績回溯到自己在各章節的真實弱點，導致弱點分析功能失準。

---

## 設計方案：三欄位溯源架構

每道題目攜帶三個追溯欄位，確保無論被複製幾次、移動到哪裡，都能精確回溯原始出處：

| 欄位 | 用途 | 範例 |
|------|------|------|
| `source_question_id` | 指向原始題目 ID | `a1b2c3...` |
| `original_subject` | 原始大分類 | 工業安全與衛生 |
| `original_chapter` | 原始章節 | 第三章 電氣安全 |

支援多層複製——即使從模擬考再複製到其他考卷，溯源鏈始終指向最初的原始題目。

---

## 成績雙向同步機制

當學員在模擬考中作答時，系統會**自動將成績同步回原始題目**，讓章節弱點分析即時反映真實能力。

### 資料流

```
學員作答「模擬考 > 第一回」的某題
        │
        ├──→ 寫入該題的作答紀錄（answer_records）
        ├──→ 更新該題的學習進度（user_question_progress）
        │
        └──→ 偵測到 source_question_id 存在
             │
             └──→ 同步更新原始題目的學習進度
                  （答對次數 +1 / 答錯次數 +1）
```

### 同步規則

| 作答場景 | 該題進度 | 原始題進度 |
|---------|---------|-----------|
| 答模擬考的複製題 | 正常更新 | 同步累加 |
| 答原始題庫的題目 | 正常更新 | —（自身即為原始題） |

**單向性設計**：答原始題不會反向影響複製題的進度，避免數據交叉污染。

### 容錯機制

同步回寫以 try-catch 包裝，即使同步失敗也不影響學員的主要作答流程，確保考試體驗不受干擾。

---

## 弱點儀表板：章節正確率總覽

基於溯源機制，弱點分析儀表板能呈現兩個維度的數據：

**維度一：考卷維度**
> 「模擬考第一回」整體正確率 75%

**維度二：章節維度（溯源統計）**
> 「第三章 電氣安全」正確率 40% — 需要加強

學員在模擬考中答錯的題目，會即時反映在原始章節的正確率上。即使同一題出現在多份考卷中，章節維度始終呈現該學員對該知識點的**累積掌握度**。

每個章節顯示三層指標：
- **精通數**：答對 3 次以上，代表長期記憶已建立
- **正確數**：至少答對 1 次，代表有基礎理解
- **總題數**：該章節的完整題目量

---

## 資料安全：每位學員的成績完全隔離

系統採用雙層防護，確保每位學員只能存取自己的作答紀錄：

**應用層**：所有資料庫操作皆綁定當前登入用戶的 `user_id`，不存在跨用戶存取的可能。

**資料庫層**：啟用 Row Level Security（RLS），即使繞過應用程式，資料庫本身也會阻擋未授權的讀寫操作。

題庫（questions）為全體共用，但作答紀錄（user_question_progress、answer_records）每位學員各自獨立、互不干擾。

---

## 設計價值

| 面向 | 價值 |
|------|------|
| 學員體驗 | 考完模擬考立即知道自己哪個章節最弱，精準複習 |
| 管理者彈性 | 自由抽題組卷，不需擔心破壞弱點分析的準確性 |
| 數據完整性 | 無論題目被複製幾次，成績永遠可追溯至知識本源 |
| 系統穩定性 | 容錯設計確保同步機制不影響核心考試流程 |
